# SVG and icon asset traps

*Recorded 2026-07-21.*

Three ways an SVG or generated icon ships silently broken. None is caught by
`npm run build`, the test suite, or looking at a thumbnail — nothing in this
project parses `public/*.svg` or inspects a generated PNG.

## 1. A doubled hyphen inside an XML comment kills the entire file

`--` is illegal inside an XML comment. An SVG containing one fails to parse, so
the browser renders a **broken image** — not a degraded icon, nothing at all.

Hit while writing an attribution comment for `public/favicon.svg` that contained
`--accent` and an `--` dash. The favicon vanished completely; `npm run build`
stayed green and all tests passed.

```bash
# Guard before committing any hand-edited SVG
python3 -c "import xml.dom.minidom as m; m.parse('public/favicon.svg'); print('ok')"
```

## 2. ImageMagick drops `fill="url(#gradient)"` and paints it pure black

A gradient-filled background renders correctly in a browser, so the source SVG
looks right — while every PNG rasterized from it ships with a black background.
Proven in isolation with two minimal files through the same renderer:

```
flat-fill     =>  srgba(18,18,26,1)   correct
gradient-fill =>  srgba(0,0,0,1)      wrong
```

Two sibling apps had shipped black icon backgrounds this way, unnoticed, because
their `icon.svg` files still previewed correctly. Use a flat `<rect fill="#…">`.

## 3. ImageMagick also drops `stroke`

Stroke-based artwork bakes out to a blank square. An icon built from strokes may
still exist as correct-looking PNGs only because they were generated by some
*other* tool — which means it is unreproducible from its own documented command.

## Verify, don't eyeball

All three failures produce a plausible-looking icon at thumbnail size.

```bash
# Background must be the intended colour, not srgba(0,0,0,1)
magick public/icon-512.png -format "%[pixel:p{8,8}]" info:

# Standard deviation near 0 means a blank square
magick public/icon-512.png -format "%[fx:standard_deviation]" info:
```

## Sizing

Derive the transform from the artwork's **measured** bounding box
(`path.getBBox()` in a browser), not an eyeballed inset. Target roughly 62% of
the canvas so the mark survives Android's circular maskable crop, and centre on
the measured centroid — source paths are often slightly off-centre in their own
viewBox.

`apple-touch-icon.png` needs `-alpha remove -alpha off`; iOS does not support
transparency in a touch icon.
